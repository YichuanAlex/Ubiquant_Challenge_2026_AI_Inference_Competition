# 技术报告 — poppycock

## 1. 系统架构

提交包采用单进程异步客户端 + 内嵌 vLLM `AsyncLLMEngine` 的简单结构：

```
submission/
├── run.sh            # 入口；激活 venv，导出保守参数，启动 client.py
├── setup.sh          # 选择 python>=3.10，建 venv，按需装 vLLM
├── requirements.txt  # 仅声明硬依赖 httpx / transformers
└── client.py         # 调度器 + 推理引擎 + 平台 HTTP 客户端
```

`client.py` 内部分四层：

1. **`PlatformClient`** — 基于 `httpx.AsyncClient` 封装 `/register /query /ask /submit`。
2. **`TaskScheduler`** — 优先队列（按 `SLA × reward × eval_timeout_s` 评分），三态显式管理 `queued / processing / completed`，**只有在有空闲 worker 时才把任务切到 `processing`**，并通过 `MAX_BUFFERED_TASKS` 做背压。
3. **`InferenceEngine`** — 包装 vLLM `AsyncLLMEngine`；提供 `generate_until / compute_loglikelihood_one / compute_rolling_one`，全部具备**硬墙时超时**。
4. **`TaskProcessor`** — 按消息逐条推理，单条失败/超时不影响其余消息，**并且无论结果如何都会调一次 `/submit`**。

主循环采用 `query → 入队 → dispatch worker → process → submit` 流水，并附带一个 `heartbeat` 协程，每 30 秒输出一次 `queue / active / completed / failed / msg_timeouts / cache` 统计，便于赛后定位问题。

## 2. 真正解决了什么问题

### 2.1 现象

之前版本"跑一段时间后只剩 `Discovered task ...` 而不再有 `[Task ...] submitted`"，连 `[Task ...] timed out` 都不打印。把单题超时降到 30 秒也跳不出来。

### 2.2 根因

`asyncio.wait_for(coro, timeout=T)` 不是"T 秒后强行结束"——它会在 T 秒到达时**取消** `coro`，然后**等待取消传播完成**。我们的旧实现里：

```python
async with self._inference_gate:
    async for request_output in self.async_engine.generate(...):
        ...
```

取消事件传到 `async for` 时，会调用 vLLM 异步生成器的 `aclose()`。在 fake 权重 / 引擎处于异常状态时，`aclose()` 自身可能永久阻塞，于是 `wait_for` 的"等待取消完成"也永久阻塞。4 个 worker 槽全部卡在这一步，主循环还在不断 `query`，但永远派不出新 worker，外观就是"假死"。

### 2.3 修复

`InferenceEngine._run_with_timeout` 改用 `asyncio.create_task` + `asyncio.wait`：

```python
inner = asyncio.create_task(self._consume_generation(...))
done, _ = await asyncio.wait({inner}, timeout=timeout_s)
if inner not in done:
    self._fire_and_forget_abort(request_id)   # 放后台执行
    inner.cancel()                            # 不 await，不等清理
    return None
```

关键点：**超时分支只 `cancel`，不 `await inner`**。即便底层 `aclose` 永远卡，主线程也已经在 `timeout_s` 秒内返回。同时 `abort` 通过 `create_task` 后台执行（自身再加 `ABORT_TIMEOUT_S` 兜底超时），不会污染推理调用栈。

## 3. 其他稳定性优化

| 类别 | 措施 |
|---|---|
| **总会提交** | `TaskProcessor` 把每条消息单独 try/except，超时/异常一律填默认值（`response=""` / `accuracy=-10.0` / `accuracy=-100.0`），最后**无条件**调 `/submit`。避免"算了一半，结果什么都没交"。|
| **per-message 超时** | 整任务预算（`TASK_TIMEOUT_S=600`）按消息数均分，并以 `MSG_TIMEOUT_S=60` 为单条上限。一条慢消息不会拖死整任务。|
| **背压** | `MAX_BUFFERED_TASKS` 限制本地队列上限，缓冲满即停止 `query`。|
| **调度无假死** | `dispatch_next` 只有在确认有空闲容量时才把任务切到 `processing`。|
| **依赖兼容** | `setup.sh` 选择 `python3.12 → 3.11 → 3.10` 任一可用；vLLM 安装失败不阻断 setup（client 自动走 mock 路径）；vLLM 版本不在 `requirements.txt` 中硬钉，避免在缺少 Blackwell 兼容轮子时直接挂掉。|
| **观测** | 心跳日志每 30 秒一行，包含队列/活跃/完成/失败/超时计数和缓存命中率，便于事后回放分析。|
| **HTTP 噪音** | 关掉 `httpx` 的 INFO 级访问日志，避免 `Discovered` 行被淹没。|

## 4. 关键参数（`run.sh` 默认值，均可由环境变量覆盖）

| 变量 | 默认 | 含义 |
|---|---|---|
| `MAX_CONCURRENT_TASKS` | 4 | 同时处理任务数上限 |
| `MAX_BUFFERED_TASKS`   | 8 | 本地缓冲（含正在处理）上限 |
| `TASK_TIMEOUT_S`       | 600 | 单任务硬墙时（与平台 `eval_timeout_s` 对齐） |
| `MSG_TIMEOUT_S`        | 60 | 单条消息硬墙时 |
| `ABORT_TIMEOUT_S`      | 3 | 调用 vLLM `abort()` 自身的超时 |
| `HEARTBEAT_INTERVAL_S` | 30 | 心跳日志间隔 |

调优顺序建议：先用默认跑通 → 把 `MAX_CONCURRENT_TASKS` 调到 6/8 → 再视情况动 `MSG_TIMEOUT_S`。出现卡顿立刻调回保守值。

## 5. 自动化验证

`tools/timeout_self_test.py`（仓库根目录，不在提交包内）用一个**模拟 vLLM 卡死的 fake 引擎**注入 `InferenceEngine`，断言：

1. `generate_until` 在 `timeout_s + 小余量` 内返回 `""`；
2. 一次卡死后，下一次调用不会被拖累；
3. `compute_loglikelihood_one` / `compute_rolling_one` 在卡死时各自返回 `-10.0 / -100.0`；
4. `TaskProcessor.process_task` 即便所有消息全部卡死也会调用 `/submit` 一次；
5. mock 路径（无 `MODEL_PATH`）依然可用、几乎无延迟。

赛机上一行运行：

```bash
cd /home/ubiservice-main && python tools/timeout_self_test.py
```

## 6. 已知取舍

- 默认 `MAX_CONCURRENT_TASKS=4`、`MSG_TIMEOUT_S=60`，在真 4×5090 + Qwen3-32B 上**吞吐有上限**；本提交以"先稳定跑完整场"优先，吞吐留给上线后视心跳数据再调。
- 超时分支不 `await` 内部任务清理，可能在极端情况下后台留有 ghost 协程；但 vLLM 的 `abort` 已经被 fire-and-forget 调度，GPU 资源由 vLLM 自身的请求管理负责回收，主流程不再受其影响。
